## Context

现有 `wiki-ingest` 已经规定：URL 在处理前要提取正文并保存为 `sources/downloaded/<domain>-<timestamp>.md`，后续 ingest 基于该本地 Markdown 副本继续进行。

但目前没有约束图片处理策略。这样会带来两个问题：

- agent 可能忽略正文强依赖的图示，导致页面缺少关键上下文
- agent 也可能过度抓取无价值图片，增加噪音、失败率和存储负担

## Goals / Non-Goals

**Goals:**
- 将 URL ingest 的图片处理定义成可选的关键资源同步，而不是默认全量抓取
- 让 spec 和 prompt 能一致约束“保留哪些图、不保留哪些图”
- 规定图片数量上限、文件命名、落盘位置、页面引用和日志行为
- 明确图片失败不影响正文 ingest 完成

**Non-Goals:**
- 当前变更不实现完整的站点资源镜像
- 当前变更不要求抓取 CSS、JS、favicon、logo 或任意媒体资源
- 当前变更不要求立即新增 CLI 下载器或对所有站点提供稳定图片解析能力

## Decisions

### D1：图片同步是可选资源同步，不是默认全量抓取

**决策**：`/wiki-ingest <url>` 以正文提取为主；只有当图片对理解正文确实必要时，才同步少量关键图片。

**原因**：文本是 ingest 主链路，图片只在少数场景构成必要上下文。把图片视为可选资源同步，可以避免 ingest 退化成低信噪比的全站采集。

**替代方案**：默认下载页面全部图片 → 噪音高、失败点多、和知识沉淀目标不匹配，放弃。

---

### D2：用启发式规则筛选关键图片

**决策**：优先保留图表、架构图、流程图、表格截图、正文强依赖的示意图；默认忽略头像、品牌横幅、装饰图、无信息量配图。

**原因**：当前阶段没有稳定的结构化图片区分器，最可行的是在 spec 和 prompt 中先给出明确的保留/丢弃规则。

**替代方案**：不设筛选规则，只让 agent 自由判断 → 漂移过大，结果不稳定，放弃。

---

### D3：同步数量受限，失败不阻塞正文

**决策**：单次 URL ingest 最多同步 3 到 5 张关键图片；任一图片下载失败时，只记录失败并继续完成文本 ingest。

**原因**：这能把资源同步控制在“补充上下文”而不是“重建页面”的级别，同时降低对网络和站点兼容性的依赖。

---

### D4：图片落盘后必须可直接被页面引用

**决策**：同步图片落到 `wiki/attachments/` 或 `sources/downloaded/assets/`；在对应 wiki 页面中使用标准 Markdown 图片引用或 Obsidian 可识别的链接插入，并附一句用途说明。

**原因**：图片的价值在于和知识页面一起被阅读，而不是单独存档。用途说明可以解释保留理由，降低“为什么同步这张图”的歧义。

## Risks / Trade-offs

**[站点结构不稳定]** → 一些页面图片懒加载、反爬或嵌套在复杂 DOM 中，agent 可能拿不到稳定链接。  
缓解：把图片同步定义为 best-effort，失败不阻塞正文 ingest。

**[图片筛选主观性]** → “是否必要”存在解释空间。  
缓解：先用明确启发式规则约束；未来如果漂移仍大，再下沉到 CLI 下载器或更稳定的解析层。

**[文件散落]** → 图片既可能进入 `wiki/attachments/`，也可能进入 `sources/downloaded/assets/`。  
缓解：spec 只要求实现至少一种稳定落盘位置，并要求页面引用必须指向本地副本。

## Open Questions

- 最终实现时应统一到 `wiki/attachments/`，还是保留 `sources/downloaded/assets/` 作为原始资源区？
- 是否需要后续新增一个 CLI 辅助能力，为 URL ingest 提供稳定的关键图片下载与文件名规范化？
- 是否需要为 Windows 默认关闭 `qmd` 的 GPU embedding 路径，并把向量索引明确降级为“有 CUDA/驱动兼容时启用，否则自动 CPU 或 text-only fallback”的可选增强能力？
